agent in the loop

A background agent that watches long-running sessions and steers them to stay on track.

Reads live execution traces at runtime Built on OpenCode + Raindrop Workshop Runs fully local
Problem

Research swarms fail in the same four ways.

Loop

A searcher chases a nonexistent file, retries the same dead query, never converges.

Duplicate

Two agents independently grind the same source — wasted tokens, wasted wall-clock.

Contradict

Subagents return conflicting claims; the parent silently accepts the wrong one.

Drift

An agent wanders off its subquestion and inflates stale or hallucinated evidence.

Today the trace is a post-mortem — you read it after the run already burned the budget. It should be a control signal, not a log.

Solution

Four patterns → four nudges, hybrid policy.

A small, fixed P→N mapping — and that mapping is the eval surface: given trace pattern P, observer emits nudge N. Cheap heuristics decide when to wake; the LLM decides what to do.

Duplicate work
Fuzzy-match search spans across agents
→
"Stop — Searcher 2 already has this." Reassign.
Contradiction
Cluster claims, flag opposing assertions
→
"Your finding conflicts with Searcher 1 — re-verify."
Stall / blocker
Span-duration thresholds + dependency state
→
"Abandon this thread, return what you have."
Drift
Semantic similarity to the assigned task
→
"Refocus on subquestion X."

Typed Intervention { action, payload, reason, confidence } — rate-limited & confidence-gated so the controller can't thrash.

Benchmarks

Measured OFF-vs-ON, not a lucky trace.

Identical workflow, model & fixture — control vs. nudged, ≥3 trials each, compared on medians. A scorer grades a machine-readable BENCHMARK_FINAL_JSON against the source-of-truth brief.
